# 노드 전달

이 문서는 현재 NW 프로젝트에서 각 node가 서로에게 어떤 작업을 수행하는지 정리한다. 실제 JSON 자료형은 [`노드 자료형.md`](./노드%20자료형.md)를 보고, 구현의 최종 기준은 root의 `IMPLEMENTATION_SPEC.md`를 우선한다.

## 전체 경로

```text
Host Simulator -> Local Agent -> Relay R1 -> Relay R2 -> Monitor
```

`Controller/UI`는 이 data plane 경로 안에서 forwarding을 수행하지 않는다. 사용자의 명령을 node에 보내고, node가 보고하는 상태를 받아 화면에 표시하는 control/status plane이다.

## 표 읽는 법

각 node는 하나의 표로 정리한다. 첫 번째 칸은 자료의 성격을 먼저 쓰고, 기술적으로 의미 있는 작업이면 괄호 안에 기술명을 함께 쓴다.

예시는 다음과 같다.

- `받는 자료`: 이전 node나 controller에서 들어오는 자료.
- `받는 자료 (duplicate suppression)`: 받은 자료에 중복 억제 기술 판단이 함께 적용되는 경우.
- `전달 자료 (EVENT forwarding)`: 다음 hop으로 event를 전달하는 경우.
- `전달 자료 (ACK)`: 이전 hop으로 처리 확인을 돌려주는 경우.
- `확인 받는 자료 (ACK)`: 보낸 자료가 처리되었는지 확인하기 위해 ACK를 받는 경우.
- `시스템 이외의 자료 (control)`: 사용자 입력, Controller/UI 제어, Controller/UI 상태 보고처럼 data plane 바깥의 자료.

## 작업 종류

- `host polling`: Local Agent가 Host Simulator에게 현재 host 상태를 물어보는 작업.
- `polling response`: Host Simulator가 Local Agent에게 현재 host 상태를 돌려주는 작업.
- `EVENT 생성`: Local Agent가 host 상태 변화나 fault를 data plane event로 바꾸는 작업.
- `EVENT forwarding`: relay가 받은 `EVENT`를 다음 hop으로 전달하는 작업.
- `ACK`: 현재 hop의 처리가 끝났음을 바로 이전 node에 돌려주는 확인 작업.
- `retry`: 다음 hop의 ACK를 받지 못했을 때 같은 `event_id`의 `EVENT`를 다시 보내는 작업.
- `duplicate suppression`: 같은 `event_id`를 이미 처리했거나 처리 중이면 중복 기록이나 중복 처리를 막는 작업.
- `control`: Controller/UI가 node의 실행 상태, fault, delay, ACK drop 같은 동작을 바꾸는 작업.
- `status report`: node가 자기 상태와 최근 송수신 사실을 Controller/UI에 보고하는 작업.

## Node별 전달 작업

### Host Simulator

Host Simulator는 data plane `EVENT`를 직접 만들거나 relay하지 않는다. Local Agent가 상태를 물어보면 현재 host 상태를 응답하고, Controller/UI가 fault 명령을 보내면 fault 상태를 만든다.

| 자료 구분 | 방향 | 의미 |
|---|---|---|
| 받는 자료 (`host polling`) | Local Agent -> Host Simulator | Agent가 event 판단을 위해 원천 host 상태를 요청한다. |
| 전달 자료 (`polling response`) | Host Simulator -> Local Agent | Host가 CPU, memory, service, latency, fault 상태를 응답한다. 이 응답은 아직 data plane `EVENT`가 아니다. |
| 시스템 이외의 자료 (`control`) | Controller/UI -> Host Simulator | `inject_fault`, `start`, `pause`, `reset`, `shutdown` 같은 제어를 수행한다. |
| 시스템 이외의 자료 (`status report`) | Host Simulator -> Controller/UI | Host의 현재 상태와 최근 polling 송수신 사실을 보고한다. |

### Local Agent

Local Agent는 Host Simulator의 상태를 주기적으로 polling하고, 그 결과를 `EVENT`로 바꿔 R1에 보낸다. 반복 fault는 같은 fault signature 동안 중복 event를 계속 만들지 않는다.

| 자료 구분 | 방향 | 의미 |
|---|---|---|
| 받는 자료 (`polling response`) | Host Simulator -> Local Agent | Agent가 `host_state`를 받아 정상 상태 변화인지 fault인지 판단한다. |
| 전달 자료 (`host polling`) | Local Agent -> Host Simulator | fault 감지와 상태 변화 감지를 위한 입력 자료를 요청한다. |
| 전달 자료 (`EVENT 생성`) | Local Agent -> Relay R1 | `HOST_STATE_UPDATE` 또는 fault `EVENT`를 만들어 R1에 보낸다. 이 지점부터 data plane hop이 시작된다. |
| 확인 받는 자료 (`ACK`) | Relay R1 -> Local Agent | R1이 downstream 처리까지 확인한 뒤 Agent에게 hop 처리 완료를 알린다. |
| 확인 받는 자료 (`ERROR`) | Relay R1 -> Local Agent | R1이 paused, pending duplicate, invalid event, delivery failure를 Agent에게 알린다. |
| 시스템 이외의 자료 (`control`) | Controller/UI -> Local Agent | Agent의 실행 상태, pending ACK, 최근 event 상태를 제어한다. |
| 시스템 이외의 자료 (`status report`) | Local Agent -> Controller/UI | polling 결과, 생성 event, R1 응답, traffic snapshot을 보고한다. |

### Relay R1

R1은 Local Agent와 R2 사이의 relay다. Local Agent가 보낸 `EVENT`를 검증하고, `event_id` 기준 duplicate suppression을 적용한 뒤 R2로 전달한다.

| 자료 구분 | 방향 | 의미 |
|---|---|---|
| 받는 자료 (`EVENT`) | Local Agent -> Relay R1 | R1이 upstream `EVENT`를 받고 필수 field를 검증한다. |
| 받는 자료 (`duplicate suppression`) | Relay R1 내부 | 같은 `event_id`가 이미 처리되었거나 처리 중이면 중복 처리를 막는다. |
| 전달 자료 (`EVENT forwarding`) | Relay R1 -> Relay R2 | R1이 받은 event를 R2로 전달하고 ACK를 기다린다. |
| 전달 자료 (`retry`) | Relay R1 -> Relay R2 | R2 ACK가 timeout되면 같은 `event_id`로 다시 보낸다. |
| 전달 자료 (`ACK`) | Relay R1 -> Local Agent | R2 처리 확인 후 Agent에게 upstream `ACK`로 hop 처리 완료를 알린다. |
| 전달 자료 (`ERROR`) | Relay R1 -> Local Agent | downstream 전달 실패나 duplicate pending 같은 실패 상태를 알린다. |
| 확인 받는 자료 (`ACK`) | Relay R2 -> Relay R1 | R2 hop 처리가 끝났음을 R1이 확인한다. 이 ACK가 와야 Agent에게 upstream `ACK`를 돌려줄 수 있다. |
| 확인 받는 자료 (`timeout`) | Relay R2 -> Relay R1 | R2에서 ACK가 오지 않으면 R1이 retry를 수행한다. |
| 시스템 이외의 자료 (`control`) | Controller/UI -> Relay R1 | delay, pause, reset, shutdown 등 relay 동작을 바꾼다. |
| 시스템 이외의 자료 (`status report`) | Relay R1 -> Controller/UI | 받은 event, 보낸 event, 받은 ACK, 보낸 ACK, retry와 duplicate 상태를 보고한다. |

### Relay R2

R2는 R1과 Monitor 사이의 relay다. R1이 보낸 `EVENT`를 검증하고 duplicate suppression을 적용한 뒤 Monitor로 전달한다.

| 자료 구분 | 방향 | 의미 |
|---|---|---|
| 받는 자료 (`EVENT`) | Relay R1 -> Relay R2 | R2가 upstream `EVENT`를 받고 필수 field를 검증한다. |
| 받는 자료 (`duplicate suppression`) | Relay R2 내부 | 같은 `event_id`가 이미 처리되었거나 처리 중이면 중복 처리를 막는다. |
| 전달 자료 (`EVENT forwarding`) | Relay R2 -> Monitor | R2가 event를 Monitor sink로 전달하고 ACK를 기다린다. |
| 전달 자료 (`retry`) | Relay R2 -> Monitor | Monitor ACK가 drop되거나 timeout되면 같은 `event_id`로 다시 보낸다. |
| 전달 자료 (`ACK`) | Relay R2 -> Relay R1 | Monitor 처리 확인 후 R1에게 upstream `ACK`로 hop 처리 완료를 알린다. |
| 전달 자료 (`ERROR`) | Relay R2 -> Relay R1 | Monitor 전달 실패를 R1에게 알린다. |
| 확인 받는 자료 (`ACK`) | Monitor -> Relay R2 | Monitor가 기록 또는 duplicate 확인을 끝냈음을 R2가 확인한다. 이 ACK가 와야 R1에게 upstream `ACK`를 돌려줄 수 있다. |
| 확인 받는 자료 (`timeout`) | Monitor -> Relay R2 | Monitor가 ACK를 보내지 않으면 R2가 retry를 수행한다. |
| 시스템 이외의 자료 (`control`) | Controller/UI -> Relay R2 | R2 delay, pause, reset, shutdown 등 relay 동작을 바꾼다. |
| 시스템 이외의 자료 (`status report`) | Relay R2 -> Controller/UI | R1/R2/Monitor 사이의 event, ACK, retry, duplicate 상태를 보고한다. |

### Monitor

Monitor는 data plane의 최종 sink다. R2가 보낸 event를 기록하고 host별 최신 상태 table을 갱신한 뒤 R2에 ACK를 반환하며, 기록한 event와 상태를 사람이 읽을 수 있는 상황판 자료로 해석해 Controller/UI의 `focus node: monitor` 화면에 제공한다.

| 자료 구분 | 방향 | 의미 |
|---|---|---|
| 받는 자료 (`EVENT`) | Relay R2 -> Monitor | Monitor가 sink `EVENT`를 최종 기록 대상으로 처리한다. |
| 받는 자료 (`duplicate suppression`) | Monitor 내부 | 이미 처리한 `event_id`면 host table과 event log를 중복 갱신하지 않는다. |
| 전달 자료 (`ACK`) | Monitor -> Relay R2 | event를 기록했거나 duplicate로 안전하게 확인했음을 sink `ACK`로 R2에 알린다. |
| 전달 자료 (`ACK drop`) | Monitor -> Relay R2 | `drop_next_ack` 상태에서는 JSON 응답을 보내지 않아 R2 timeout과 retry를 유도한다. |
| 시스템 이외의 자료 (`control`) | Controller/UI -> Monitor | `drop_next_ack`로 다음 ACK 한 번을 의도적으로 생략하게 한다. |
| 시스템 이외의 자료 (`status report`) | Monitor -> Controller/UI | 기록 event, host state table, duplicate/out-of-order count, ACK 상태를 보고한다. |
| 시스템 이외의 자료 (`human-readable display`) | Monitor -> Controller/UI | event와 host state를 `처리 경로`, `현재 상황`, `Host 최신 상태`, `최근 알림/이벤트`, `전달 건강도`, `확인 응답 / 재시도` 중심의 터미널-safe TUI 상황판으로 표시할 수 있게 한다. 넓은 화면에서는 안전한 박스형 배치를 쓰고, 좁은 화면에서는 섹션/plain line fallback을 사용한다. |

### Controller/UI

Controller/UI는 사람이 입력한 text command를 해석해 node에 control message를 보내고, node가 보내는 status report를 모아 화면에 표시한다. data plane forwarding node가 아니므로 아래 표는 control/status plane 기준으로 읽는다.

| 자료 구분 | 방향 | 의미 |
|---|---|---|
| 받는 자료 (`사용자 입력`) | 사용자 -> Controller/UI | `fault`, `ackdrop`, `delay`, `focus`, `quit` 같은 text command를 입력한다. |
| 받는 자료 (`status report`) | node -> Controller/UI | 각 node 상태와 traffic snapshot을 모아 live/stale/offline 표시를 계산한다. |
| 전달 자료 (`control`) | Controller/UI -> Host Simulator | fault 상태를 만들도록 Host Simulator에 지시한다. |
| 전달 자료 (`control`) | Controller/UI -> Relay R1/R2 | relay delay나 실행 상태를 바꿔 ACK/retry 흐름을 관찰하게 한다. |
| 전달 자료 (`control`) | Controller/UI -> Monitor | Monitor의 다음 ACK를 한 번 생략하게 한다. |
| 전달 자료 (`control`) | Controller/UI -> 전체 또는 특정 node | `start`, `pause`, `reset`, `shutdown`을 전달한다. |
| 확인 받는 자료 (`status report`) | node -> Controller/UI | node는 control 명령에 별도 data-plane `ACK`를 보내지 않고, 이후 상태 보고로 제어 결과를 관찰하게 한다. |
| 시스템 이외의 자료 (`UI local state`) | Controller/UI 내부 | `focus`, `overview`, `focus all`은 node에 `CONTROL`을 보내지 않고 UI 내부 관찰 대상만 바꾼다. |
| 시스템 이외의 자료 (`liveness`) | Controller/UI 내부 | `STATUS.state`와 별도로 수신 시각을 기준으로 `live`, `stale`, `offline`을 계산한다. |

## 대표 전달 흐름

### 정상 상태 변화

```text
Local Agent -> Host Simulator: host polling
Host Simulator -> Local Agent: polling response
Local Agent -> R1: EVENT 생성 및 첫 hop 전달
R1 -> R2: EVENT forwarding, ACK 대기
R2 -> Monitor: EVENT forwarding, ACK 대기
Monitor -> R2: sink ACK
R2 -> R1: upstream ACK
R1 -> Local Agent: upstream ACK
```

### Fault 이벤트

```text
Controller/UI -> Host Simulator: fault injection control
Host Simulator: fault 상태 생성
Local Agent -> Host Simulator: host polling
Host Simulator -> Local Agent: fault가 반영된 polling response
Local Agent -> R1: fault EVENT 생성 및 첫 hop 전달
R1 -> R2 -> Monitor: EVENT forwarding
Monitor -> R2 -> R1 -> Local Agent: hop-by-hop ACK
```

### ACK drop / retry / duplicate suppression

```text
Controller/UI -> Monitor: drop_next_ack control
R2 -> Monitor: EVENT forwarding
Monitor: event 기록 후 ACK 1회 미전송
R2: ACK timeout 후 같은 event_id로 retry
Monitor: duplicate suppression으로 중복 기록 방지
Monitor -> R2: duplicate 확인 후 ACK 반환
R2 -> R1 -> Local Agent: upstream ACK 반환
```
