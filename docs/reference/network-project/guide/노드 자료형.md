# 노드 자료형

이 문서는 현재 NW 프로젝트에서 각 node가 주고받는 자료형을 정리한다. 사용자가 화면에서 보이는 JSON 자료를 이해하기 위한 guide이며, 상세 구현 기준은 root의 `IMPLEMENTATION_SPEC.md`를 우선한다.

아래 JSON은 줄 단위 TCP 메시지로 실제 전송되는 자료 형식을 보여주는 예시다. `timestamp`와 상태 값은 실행 시점마다 바뀌지만, field 구조는 현재 구현 기준을 따른다.

## 전체 흐름

기본 data plane은 아래 순서로 흐른다.

```text
Host Simulator -> Local Agent -> Relay R1 -> Relay R2 -> Monitor
```

`Controller/UI`는 data plane forwarding node가 아니다. 사용자의 명령을 각 node에 보내고, node가 보고하는 상태를 받아 화면에 표시하는 control/status plane이다.

통신은 TCP 연결 위에서 줄 단위 JSON 메시지로 이루어진다. 한 메시지는 JSON 한 줄로 encode되어 전송된다.

노드 사이에서 어떤 작업이 이루어지는지의 흐름 설명은 [`노드 전달.md`](./노드%20전달.md)를 본다.

## Node별 송수신 자료형

### Host Simulator

Host Simulator는 가상의 host 관측값을 만든다. 정상 상태에서는 CPU, memory, service, latency 값이 tick마다 바뀌고, 사용자가 fault 명령을 보내면 일정 시간 동안 fault 주입에 맞는 비정상 관측값을 만든다. `fault_mode` 같은 의미 판단은 Host가 아니라 Local Agent가 작성하는 EVENT payload에만 둔다.

Host Simulator는 R1/R2/Monitor로 `EVENT`를 직접 보내지 않는다. Local Agent가 상태를 물어볼 때 raw `host_state` 응답을 돌려준다.

#### 받음: Local Agent -> Host Simulator

```json
{
  "kind": "get_host_state"
}
```

#### 보냄: Host Simulator -> Local Agent

```json
{
  "kind": "host_state",
  "host_state": {
    "host_id": "host-1",
    "cpu_usage": 22,
    "memory_usage": 45,
    "service_state": "UP",
    "latency_ms": 30,
    "last_update_time": "2026-04-29T12:00:00+00:00"
  }
}
```

#### 받음: Controller/UI -> Host Simulator

```json
{
  "msg_type": "CONTROL",
  "command": "inject_fault",
  "target": "host-simulator",
  "params": {
    "fault_type": "CPU_SPIKE",
    "duration_sec": 6
  },
  "control_token": "demo123",
  "timestamp": "2026-04-29T12:00:00+00:00"
}
```

```json
  "target": "host-simulator",
  "params": {},
```

### Local Agent

Local Agent는 Host Simulator를 주기적으로 polling한다. 받은 host 상태에서 fault를 감지하고, 정상 상태 변화나 fault를 `EVENT`로 만들어 R1에 보낸다.

현재 기준에서는 fault가 없어도 host 상태 변화가 있으면 `HOST_STATE_UPDATE` 이벤트를 만든다. fault가 반복되는 동안에는 같은 fault event를 매 tick마다 중복 생성하지 않는다.

#### 보냄: Local Agent -> Host Simulator

```json
{
  "kind": "get_host_state"
}
```

#### 받음: Host Simulator -> Local Agent

```json
{
  "kind": "host_state",
  "host_state": {
    "host_id": "host-1",
    "cpu_usage": 22,
    "memory_usage": 45,
    "service_state": "UP",
    "latency_ms": 30,
    "last_update_time": "2026-04-29T12:00:00+00:00"
  }
}
```

#### 보냄: Local Agent -> Relay R1

```json
{
  "msg_type": "EVENT",
  "event_id": "evt-host-1-1",
  "seq_no": 1,
  "host_id": "host-1",
  "agent_id": "agent-1",
  "event_type": "HOST_STATE_UPDATE",
  "severity": "INFO",
  "timestamp": "2026-04-29T12:00:00+00:00",
  "payload": {
    "cpu": 22,
    "memory": 45,
    "service_state": "UP",
    "latency_ms": 30,
    "fault_mode": "NORMAL"
  }
}
```

fault 상태에서는 같은 형식에서 `event_type`, `severity`, `payload` 값만 바뀐다.

```json
{
  "msg_type": "EVENT",
  "event_id": "evt-host-1-2",
  "seq_no": 2,
  "host_id": "host-1",
  "agent_id": "agent-1",
  "event_type": "CPU_SPIKE",
  "severity": "WARN",
  "timestamp": "2026-04-29T12:00:02+00:00",
  "payload": {
    "cpu": 96,
    "memory": 48,
    "service_state": "UP",
    "latency_ms": 36,
    "fault_mode": "CPU_SPIKE"
  }
}
```

#### 받음: Relay R1 -> Local Agent

정상 처리 시 R1은 Local Agent에 hop-by-hop `ACK`를 돌려준다.

```json
{
  "msg_type": "ACK",
  "ack_for": "evt-host-1-1",
  "from_node": "r1",
  "timestamp": "2026-04-29T12:00:03+00:00"
}
```

R1이 일시정지, pending 중복, 전달 실패, invalid event를 만나면 `ERROR`를 돌려줄 수 있다.

```json
{
  "msg_type": "ERROR",
  "reason": "delivery_failed",
  "event_id": "evt-host-1-1",
  "node_id": "r1"
}
```

### Relay R1

R1은 Local Agent에게서 `EVENT`를 받는 첫 relay다. event 필수 필드를 검증하고, `event_id` 기준 중복 여부를 확인한 뒤 R2로 전달한다. R2의 `ACK`를 기다리며 timeout이 나면 최대 3회까지 retry한다.

R1은 end-to-end ACK를 만들지 않는다. R2가 처리했다는 hop-by-hop ACK를 받은 뒤 Local Agent에 자기 ACK를 돌려준다.

#### 받음: Local Agent -> Relay R1

```json
{
  "msg_type": "EVENT",
  "event_id": "evt-host-1-1",
  "seq_no": 1,
  "host_id": "host-1",
  "agent_id": "agent-1",
  "event_type": "HOST_STATE_UPDATE",
  "severity": "INFO",
  "timestamp": "2026-04-29T12:00:00+00:00",
  "payload": {
    "cpu": 22,
    "memory": 45,
    "service_state": "UP",
    "latency_ms": 30,
    "fault_mode": "NORMAL"
  }
}
```

#### 보냄: Relay R1 -> Relay R2

R1은 받은 `EVENT`를 같은 자료 형식으로 R2에 보낸다.

```json
{

  "event_id": "evt-host-1-1",
  "seq_no": 1,
```

### Relay R2

R2는 R1과 같은 relay 로직을 수행하지만 downstream 대상이 Monitor다. R1에게서 `EVENT`를 받고 Monitor로 전달한 뒤 Monitor의 `ACK`를 기다린다.

Monitor가 ACK를 드롭하거나 응답이 늦으면 R2가 retry를 수행한다. 같은 `event_id`가 다시 들어오면 duplicate로 처리하고 필요하면 ACK를 재전송한다.

### Monitor

Monitor는 data plane의 최종 sink다. R2가 보낸 `EVENT`를 기록하고, host별 최신 상태 table을 갱신한 뒤 R2에 `ACK`를 반환한다. 또한 이 상태 자료를 Controller/UI의 `focus node: monitor` 상황판에서 사람이 읽는 처리 경로, 현재 상황, host 상태, 알림, 전달 건강도, 확인 응답/재시도 상태로 해석할 수 있게 보고한다.

`ackdrop` 명령을 받으면 다음 ACK를 한 번 의도적으로 보내지 않는다. 이때 R2는 timeout 후 같은 `event_id`로 retry하고, Monitor는 duplicate를 기록 중복 없이 처리한다.

#### 받음: Relay R2 -> Monitor

```json
{
  "msg_type": "EVENT",
  "event_id": "evt-host-1-1",
  "seq_no": 1,
  "host_id": "host-1",
  "agent_id": "agent-1",
  "event_type": "HOST_STATE_UPDATE",
  "severity": "INFO",
  "timestamp": "2026-04-29T12:00:00+00:00",
  "payload": {
    "cpu": 22,
    "memory": 45,
    "service_state": "UP",
    "latency_ms": 30,
    "fault_mode": "NORMAL"
  }
}
```

#### 보냄: Monitor -> Relay R2

```json
{
  "msg_type": "ACK",
  "ack_for": "evt-host-1-1",
  "from_node": "monitor",
  "timestamp": "2026-04-29T12:00:02+00:00"
}
```

`drop_next_ack`가 켜져 있으면 Monitor는 이 응답을 한 번 보내지 않는다. 이때 전송되는 JSON은 없고, R2 쪽에서 timeout이 발생한다.

#### 받음: Controller/UI -> Monitor

```json
{
  "msg_type": "CONTROL",
  "command": "drop_next_ack",
  "target": "monitor",
  "params": {},
  "control_token": "demo123",
  "timestamp": "2026-04-29T12:00:00+00:00"
}
```

## 메시지별 핵심 규칙

- `EVENT`는 Agent가 만들고 relay chain을 따라 Monitor까지 전달되는 data plane 메시지다.
- `ACK`는 각 hop에서 바로 이전 node에게 돌려주는 수락/처리 응답이다.
- `CONTROL`은 Controller/UI가 node를 제어할 때 보내는 control plane 메시지다.
- `STATUS_REPORT`는 shared token이 있을 때 raw `STATUS`를 감싸는 wrapper다.
- 각 node의 `detail.traffic`은 그 node가 자기 시점에서 본 최근 송수신 사실이다. Controller가 임의로 재구성한 data-plane truth가 아니다.
